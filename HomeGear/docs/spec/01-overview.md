# 1. アプリ概要

## 1.1 目的

家庭内にある以下のような機器を一元管理するスマホアプリ。

- 家電（テレビ・冷蔵庫・洗濯機など）
- IT 機器（PC・NAS・周辺機器）
- ネットワーク機器（ルーター・スイッチ・無線 AP）
- スマートホーム機器（照明・センサー・ハブ）
- 防犯カメラ・自宅サーバー機器・工具・生活設備 など

単なる所有物リストではなく、以下を 1 画面に集約した「家庭内機器台帳」を目指す:

- 何を持っているか / どこに置いてあるか / いつ買ったか
- 保証期間 / 取扱説明書・保証書・購入履歴 URL
- メンテナンスや修理履歴 / 型番・シリアル番号・付属品・消耗品
- PC や NAS のスペック / ネットワーク機器の IP アドレス・管理 URL
- AI で調べた機器情報を、コピペで取り込む

## 1.2 想定利用者

- 家庭内の大人ユーザー（主に管理者 1 名が登録・更新し、家族も閲覧・編集できる）
- スマホ操作を最重視（Android / iPhone 両対応）
- 受入時点では家族招待機能は未実装。手動で `household_members` に行を追加して共有する

## 1.3 設計思想

CLAUDE.md にも記載のルールが前提となっている。重要なものを再掲する。

- **データは家庭単位で完全分離**: 全主要テーブルに `household_id` を持たせ、API は必ず `req.auth.householdId` で絞り込む
- **パスワード・APIキー・秘密鍵は機器情報として保存しない**: ネットワーク情報には保管場所メモのみ（例: 「1Password に保存」）
- **論理削除を基本にする**: 機器は `POST /api/devices/:id/dispose` で `status = 'disposed'` + `deleted_at` を設定するのが原則。`DELETE` は誤登録の救済用
- **AI 取込のデフォルトは「空欄項目のみ反映」**: ユーザーが明示的に「上書きする」を選ばない限り既存値は触らない
- **スマホ片手操作優先**: フォームはセクション分け、必須入力は最小限

## 1.4 MVP で実装済みの機能

| # | 機能 |
|---|---|
| 1 | メールアドレス + パスワード認証（JWT 発行・有効期限 7 日） |
| 2 | 新規登録時に家庭を自動作成し owner として紐づけ、初期カテゴリ 8 件・初期設置場所 11 件を投入 |
| 3 | 機器の登録・一覧・詳細・編集 |
| 4 | キーワード検索 / カテゴリ・設置場所・ステータス絞り込み |
| 5 | 並び順切替（作成日 / 更新日 / 名前 / 保証期限 / 購入日 / 次回メンテ予定） |
| 6 | 廃棄（論理削除）と廃棄済みフィルタ |
| 7 | 子テーブル CRUD: スペック・リンク・メンテナンス履歴・修理履歴・消耗品・付属品・ネットワーク情報 |
| 8 | 修理ステータス変更で機器ステータスを自動同期（broken / repairing / in_use / disposed） |
| 9 | AI 調査用プロンプト生成 + JSON 貼付 + プレビュー + 反映（既存値温存がデフォルト） |
| 10 | ダッシュボード（機器数・保証期限・メンテ予定・故障中・情報不足・最近更新） |
| 11 | カテゴリ管理・設置場所管理 |
| 12 | JSON エクスポート（家庭内全データ） |

## 1.6 既知の運用上の注意

- **日付入力は YYYY-MM-DD の手入力**（DatePicker なし）
- **写真は URL 文字列のみ**（カメラ連携・画像アップロードはなし）
- **実機/Android エミュレータからの API 接続は自動推論**:
  - dev モードでは Expo の `hostUri` から開発 PC の IP を取得して `http://<host>:4000` を使用
  - Android エミュレータは `10.0.2.2:4000` にフォールバック
  - 本番ビルド等で固定 URL を使いたい場合は `apps/mobile/app.json` の `extra.apiBaseUrl` を上書きする
- **Web (Expo Web) はテスト用途のみ**を想定。SecureStore は `localStorage` 経由にフォールバックする
