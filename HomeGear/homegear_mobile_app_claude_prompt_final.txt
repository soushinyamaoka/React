- 回答とコード内コメントは日本語を基本にしてください。

# 家庭内機器管理スマホアプリ 仕様書

## 1. アプリの目的

家庭内にある家電、IT機器、ネットワーク機器、スマートホーム機器、防犯カメラ、自宅サーバー関連機器、工具、生活設備などを一元管理するスマホアプリを作成する。

単なる所有物リストではなく、以下をまとめて管理できる「家庭内機器台帳」として使いたい。

- 何を持っているか
- どこに置いてあるか
- いつ買ったか
- 保証期間はいつまでか
- 取扱説明書・保証書・購入履歴はどこにあるか
- メンテナンスや修理履歴はどうなっているか
- 型番・シリアル番号・付属品・消耗品は何か
- PCやNASなどのスペックは何か
- ネットワーク機器やスマート家電のIPアドレス、管理URL、連携情報は何か
- AIで調べた機器情報を、アプリ側にコピペで取り込めるようにする

## 2. 想定利用者

- 家庭内の大人ユーザー
- 主に管理者1名が登録・更新する
- 家族も同じ台帳を閲覧・編集できる
- スマホから使いやすいことを最重視する
- Android / iPhoneの両方で利用できるようにする

## 3. 技術構成

### 3.1 スマホアプリ

- Expo
- React Native
- TypeScript
- Expo Router または React Navigation
- React Hook Formなどのフォームライブラリを使ってもよい
- Zodなどでバリデーションしてもよい
- TanStack QueryなどでAPI通信・キャッシュ管理をしてもよい

### 3.2 バックエンドAPI

以下のいずれかで実装する。

- Node.js + Express
- NestJS
- Fastify
- Hono
- Next.js API Routesではなく、スマホアプリから使う独立APIとして設計するのが望ましい

### 3.3 データベース

- PostgreSQLを第一候補とする
- PrismaなどのORMを使ってもよい
- 将来的にさくらVPS、自宅サーバー、または他のVPS上で運用できる構成にする

### 3.4 端末内DBについて

- 端末内SQLiteをメインDBにはしない
- 家族間でデータを共有するため、正式データはサーバー側PostgreSQLに保存する
- SQLiteを使う場合は以下の用途に限定する
  - オフラインキャッシュ
  - 一時保存
  - 下書き保存
  - 通信失敗時の再送キュー

MVPでは端末内SQLiteは必須ではない。
まずはオンライン前提のスマホアプリとして作成する。

### 3.5 配布方針

開発中：
- Expo GoまたはDevelopment Buildで確認する

家族利用・長期運用：
- Android: EAS BuildのInternal Distribution
- iPhone: TestFlight
- 必要に応じてDevelopment Buildを使う

Expo Goによる長期共有は仕様変更やログイン制限の影響を受ける可能性があるため、家族で長期利用する場合はEAS BuildやTestFlightを前提にする。

## 11. 機器スペック管理機能

機器ごとにスペック情報を記録できるようにする。

PC、スマートフォン、タブレット、NAS、防犯カメラ、ネットワーク機器などは、機器カテゴリによって記録したいスペック項目が大きく異なる。

そのため、devicesテーブルにcpuやmemoryなどの固定カラムを直接追加するのではなく、device_specsという別テーブルに自由項目として保持する。

### 入力方式

MVPでは、カテゴリ別の固定フォームではなく、以下のような自由入力形式とする。

- スペック項目名
- 値
- 単位
- メモ
- 表示順

### PCで記録する例

- CPU：Intel Core i5 / Ryzen 5など
- メモリ：16GB
- ストレージ：512GB SSD
- GPU：NVIDIA GeForceなど
- OS：Windows 11 Pro
- 画面サイズ：14インチ
- 解像度：1920x1080
- Office有無：あり
- 増設履歴：メモリを8GBから16GBに増設

### スマートフォン・タブレットで記録する例

- 容量：256GB
- カラー：ブラック
- SIM種別：eSIM
- キャリア：docomoなど
- OSバージョン：Android / iOS

### NASで記録する例

- ベイ数：2
- 搭載HDD：4TB x 2
- RAID構成：RAID1
- CPU
- メモリ

### 防犯カメラで記録する例

- 解像度：2K / 4K
- 電源方式：バッテリー / 常時給電
- microSD最大容量：512GB
- Wi-Fi対応：2.4GHzのみ / 5GHz対応

### 仕様

- 1つの機器に複数のスペック情報を登録できる
- 項目名は自由入力できる
- 表示順を指定できる
- AI調査結果から取り込んだスペックもdevice_specsに登録する
- 将来的にはカテゴリごとのスペックテンプレートを追加できる設計にする
- MVPでは汎用的な自由入力欄として実装する

## 21. AI調査結果取り込み機能

機器を最初に登録する際、メーカー名・型番などをもとに、ChatGPT、Claude、GeminiなどのAIに機器情報を調査してもらい、その結果をアプリに貼り付けて反映できるようにする。

アプリ内でAI APIを直接呼び出す方式も考えられるが、MVPでは以下の理由により見送る。

- API利用料金が発生する可能性がある
- トークン制限を考慮する必要がある
- 各AIサービスごとのAPIキー管理が必要になる
- 家庭内利用のMVPとしては仕組みが重くなる

そのため、MVPでは「AIチャットの出力をコピペで取り込む方式」を採用する。

### 基本方針

- アプリ側でAIに渡す調査用プロンプトを生成する
- ユーザーがそのプロンプトを任意のAIチャットに貼り付ける
- AIは指定されたJSON形式で機器情報を出力する
- ユーザーがAIの出力JSONをアプリに貼り付ける
- アプリ側でJSONを解析し、登録内容のプレビューを表示する
- ユーザーが確認してから機器情報に反映する

### 対象情報

AI調査結果から、以下の情報を取り込めるようにする。

- メーカー
- 正式な製品名
- 型番
- カテゴリ
- 製品概要
- 主なスペック
- 取扱説明書URL
- メーカー公式ページURL
- サポートページURL
- 消耗品情報
- 付属品情報
- 注意事項
- メンテナンス目安

### 取り込み手順

1. ユーザーが機器名、メーカー、型番など分かる範囲で入力する
2. アプリがAI調査用プロンプトを生成する
3. ユーザーがプロンプトをコピーする
4. ChatGPT、Claude、Geminiなど任意のAIに貼り付ける
5. AIがJSON形式で回答する
6. ユーザーがJSONをアプリに貼り付ける
7. アプリが内容を解析する
8. 反映前プレビューを表示する
9. ユーザーが確認して登録・更新する

### 重要な仕様

- AIの回答は正しいとは限らないため、即時反映せず必ずプレビュー画面を挟む
- 既存の入力値を勝手に上書きしない
- 初期設定では、空欄項目のみ補完する
- 上書きする場合は項目ごとにユーザーが選択する
- URLは形式チェックを行う
- 不明な項目は無理に埋めず、nullにする
- AIが返したスペック情報はdevice_specsに登録する
- AIが返したリンク情報はdevice_linksに登録する
- AIが返した消耗品情報はconsumablesに登録する
- AIが返した付属品情報はaccessoriesに登録する

### AI調査用プロンプト生成機能

アプリ側で、以下のようなプロンプトを生成できるようにする。

## 26. バリデーション仕様

### 機器登録

- 機器名は必須
- URL項目はURL形式であること
- 購入価格は0以上の数値
- 保証終了日は保証開始日以降
- 購入日は未来日でも許可するが警告を出す

### スペック

- spec_nameは必須
- spec_valueは任意
- unitは任意
- sort_orderは数値

### メンテナンス履歴

- 実施日は必須
- 次回予定日は実施日以降が望ましい

### 修理履歴

- 発生日は必須
- 完了日は発生日以降が望ましい

### AI取り込み

- JSONとして解析できること
- 想定スキーマと異なる場合はエラーまたは警告を出す
- URLはURL形式チェックを行う
- 不明項目はnullまたは空欄として扱う
- 既存データは初期状態では上書きしない
- 取り込み前に必ずプレビューを表示する

## 27. 非機能要件

### 操作性

- スマホで片手操作しやすいこと
- 登録フォームは長すぎないようにセクション分けすること
- 必須入力を少なくすること
- 検索しやすいこと
- ダッシュボードで今見るべき情報が分かること
- 入力画面ではキーボードに隠れないようにすること
- Android / iPhoneの両方で自然に使えるUIにすること

### パフォーマンス

- 家庭内利用として、登録機器数は最大500件程度を想定
- 一覧表示は検索・絞り込み込みで快適に動作すること
- スマホアプリ側ではAPIレスポンスを適度にキャッシュしてよい

### セキュリティ

- ログイン機能を設ける
- パスワードはハッシュ化して保存する
- パスワード、APIキー、秘密鍵はMVPでは機器情報として保存しない
- 認証情報の保存場所メモのみ登録可能とする
- APIはHTTPS前提にする
- 管理画面URLなどは不用意に外部公開しない
- AI APIキーはMVPでは扱わない

### バックアップ

- JSONエクスポート機能を用意する
- PostgreSQLのバックアップを取りやすい構成にする
- 将来的には定期バックアップに対応する
