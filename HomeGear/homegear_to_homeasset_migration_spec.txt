あなたは優秀なフルスタックエンジニアです。
既存の「家庭内機器管理アプリ」から、新しい「家庭内資産・住宅設備管理アプリ HomeAsset」へデータ移行を行いたいです。
以下の仕様に沿って、移行方針・DB差分・マッピング・移行スクリプト・検証手順・ロールバック手順を提示し、実装してください。

# HomeGear から HomeAsset へのデータ移行仕様書

## 1. 背景

既に前回仕様で、家庭内機器管理アプリを作成済みである。

前回アプリでは、主に以下の考え方で設計されている。

- 管理対象は「機器」
- 中心テーブルは devices
- スペック情報は device_specs
- リンク情報は device_links
- メンテナンス履歴、修理履歴、消耗品、付属品、ネットワーク情報は device_id に紐づく

今回、新仕様として、家電・IT機器だけでなく、住宅設備や建物部位も管理対象に含めたい。

新仕様では、中心概念を「機器」ではなく「家庭内資産」に広げる。

- devices → home_assets
- device_specs → asset_specs
- device_links → asset_links
- device_id → asset_id
- 家電・PC・ルーターだけでなく、給湯器、システムキッチン、外壁、屋根、浴室、トイレなども管理対象にする

そのため、既存データを新仕様のDBへ移行する必要がある。

## 2. 移行の目的

既存アプリに登録済みの機器データを失わず、新しいHomeAsset仕様へ移行する。

移行後は、既存の機器データがすべて「家庭内資産」として扱えるようにする。

移行対象：

- 機器本体情報
- カテゴリ
- 設置場所
- スペック情報
- リンク情報
- メンテナンス履歴
- 修理履歴
- 消耗品
- 付属品
- ネットワーク情報
- AI取り込み履歴がある場合は可能な範囲で移行

#### 方式A: DB直接移行

MVPでは、実装しやすく安全な方式として、方式Bを優先してもよい。
ただし、データ量が少なく、DB構造が明確であれば方式Aでもよい。

## 4. 移行方針

### 4.1 基本方針

- 移行前に必ずバックアップを取る
- 移行処理は一括実行できるようにする
- 移行処理はdry-runできるようにする
- 移行処理は冪等性を意識する
- 移行後に件数チェックを行う
- 旧DBは削除せず、一定期間保持する
- 旧IDと新IDの対応表を残す
- 既存データを勝手に破棄しない
- 不明な値はnullまたはmemoへ退避する
- 旧仕様で「devices」として登録されていたものは、原則としてasset_type = device として移行する

### 4.2 安全性

移行スクリプトには以下のモードを用意する。

- dry-run
  - 実際にはDBへ書き込まず、移行予定件数・マッピング・警告だけ表示する
- execute
  - 実際に移行する
- verify
  - 移行後の件数・整合性を確認する

### 4.3 冪等性

同じ移行を複数回実行しても、重複登録されにくいようにする。

そのため、以下のような移行管理テーブルを新DBに作成することを推奨する。

## 5. 移行管理テーブル

### migration_mappings

旧テーブルのIDと新テーブルのIDの対応を記録する。

#### カラム案

- id
- migration_name
- source_table
- source_id
- target_table
- target_id
- created_at

#### ユニーク制約

- migration_name
- source_table
- source_id
- target_table

### migration_logs

移行実行ログを記録する。

#### カラム案

- id
- migration_name
- mode
- status
- started_at
- finished_at
- summary_json
- error_message

### 仕様

- migration_mappingsを見て、既に移行済みのデータは再作成しない
- dry-runではmigration_mappingsには書き込まない
- executeで成功した場合のみmigration_mappingsへ書き込む
- 失敗時はmigration_logsにエラーを記録する

7. categories 移行
8. locations 移行
9. devices → home_assets 移行
10. device_specs → asset_specs 移行

13. repair_records 移行
14. consumables 移行
15. network_infos 移行

17. ai_import_logs 移行
18. 件数チェック

21. 旧DBを保管


### 21.2 新アプリ側インポート処理

- devices → home_assets

### 21.3 インポート画面

新アプリに移行用の管理画面または一時的なCLIを用意する。

#### CLI例

```bash
pnpm migrate:homegear --input ./exports/homegear-export.json --execute
```

## 24. ロールバック仕様

移行に失敗した場合に備えて、ロールバック手順を用意する。

### 24.1 原則

- 移行前に旧DBと新DBのバックアップを取る
- execute前にdry-runを必ず行う
- execute中にエラーが出た場合は、トランザクションをロールバックする
- 大量データを分割移行する場合も、テーブル単位または全体単位でロールバックできるようにする

### 24.2 PostgreSQLバックアップ例

```bash
pg_dump "$OLD_DATABASE_URL" > backup_old_before_migration.sql
pg_dump "$NEW_DATABASE_URL" > backup_new_before_migration.sql
```

### 24.3 復元例

```bash
psql "$OLD_DATABASE_URL" < backup_old_before_migration.sql
psql "$NEW_DATABASE_URL" < backup_new_before_migration.sql
```

## 25. エラー・警告仕様

移行中に以下のような問題があれば警告として出力する。

- category_idが存在しない
- location_idが存在しない
- device_idに対応するasset_idがない
- statusが変換表にない
- link_typeが変換表にない
- URL形式が不正
- 日付形式が不正
- required項目が空
- 同じ旧IDが既に移行済み

警告の扱い：

- 致命的でないものは移行を継続
- 参照先が存在しないなど重大なものは該当レコードをスキップし、ログに記録
- スキップ件数を最後に表示
- 可能であればCSVまたはJSONでエラーレポートを出力する

## 27. 受け入れ条件

### 27.1 dry-run

- dry-runで実DBに書き込まない
- 移行予定件数が表示される
- カテゴリごとのasset_type推測結果が表示される
- スキップ予定のレコードが表示される
- 警告が表示される

### 27.2 execute

- 旧devicesが新home_assetsへ移行される
- 旧device_specsが新asset_specsへ移行される
- 旧device_linksが新asset_linksへ移行される
- 各関連テーブルのdevice_idがasset_idへ正しく変換される
- migration_mappingsに旧IDと新IDの対応が保存される
- 途中で致命的エラーが起きた場合はロールバックされる

### 27.3 verify

- 旧件数と新件数を比較できる
- 関連データの参照整合性を確認できる
- 問題があればエラーレポートを出力する

### 27.4 アプリ確認

移行後のHomeAssetアプリで以下が確認できる。

- 既存機器が資産一覧に表示される
- 既存機器の詳細が開ける
- スペックが表示される
- リンクが表示される
- メンテナンス履歴が表示される
- 修理履歴が表示される
- 消耗品が表示される
- 付属品が表示される
- ネットワーク情報が表示される
- 保証期限・ステータスが正しい
